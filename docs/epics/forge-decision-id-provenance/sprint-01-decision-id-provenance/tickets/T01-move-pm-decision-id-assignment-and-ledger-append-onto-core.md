# T01 — Move PM decision-id assignment and ledger append onto Core

## Scope

Close both decision-id seams so the path matches ForgeGate doctrine: **Core assigns the authoritative field,
the agent echoes it, Core validates the echo, Core records the decision.** Today the orchestrator computes the
id in prose and hand-writes the ledger JSON; C4's `appendDecision` guard exists but no CLI command calls it.

Two changes:

1. **Assignment into `forge dispatch pm`.** In `src/cli/run.ts` (the `dispatch` command path, `runDispatch`),
   when the role is `pm`: read `<epic>/.forge/decisions-ledger.json` through Core
   (`readDecisionsLedger` + a `DecisionsLedgerIo` seam), compute the next id with `nextDecisionId(existing)`,
   and pass that Core-computed id into the (still pure) PM prompt builder. The `--assigned-decision-id` flag
   becomes an **optional cross-check**: if omitted, Core's computed id is used; if supplied and it disagrees
   with Core's computed id, fail with `DECISION_ID_PROVENANCE_MISMATCH` and render no PM prompt. The flag is
   never the source of truth.

2. **`forge ledger append` command.** Add a small CLI-facing adapter
   (`src/orchestrator/ledger-cli.ts`) exposing the command
   `forge ledger append <epic> --decision-id <id> --ticket <ticket> --branch <branch>`. It builds a
   `LedgerEntry` (generating `ts` internally as an ISO timestamp), calls the existing `appendDecision` through
   an injected `DecisionsLedgerIo` seam against `<epic>/.forge/decisions-ledger.json`, and surfaces the
   result. Route it from `src/cli/run.ts` the same way `run-report write` and `guard paths` are routed. This
   puts C4's `LEDGER_SEQUENCE_INVALID` guard on the live path and removes the hand-authored ledger append.

## Required behavior

### Assignment (`dispatch pm`)
- Locate the ledger at `<epic>/.forge/decisions-ledger.json` from the dispatch epic path.
- Absent ledger → empty → `nextDecisionId([])` = `D-001`.
- Malformed ledger → `LEDGER_INVALID`, fail-closed (do not assign, do not render a PM prompt).
- Compute `nextDecisionId(existing.map(d => d.decision_id))` — unchanged allocator semantics.
- Render the computed id into the PM prompt's authoritative section (existing `renderAssignedDecisionId`
  path); the existing null-throw guard stays as defense-in-depth.
- `--assigned-decision-id` optional: omitted → use Core's computed id; supplied + equal → succeed; supplied +
  unequal → `DECISION_ID_PROVENANCE_MISMATCH`, no render.
- `buildPmDispatch` stays a **pure** assembler — no file IO inside it; the ledger read happens in the CLI
  orchestration layer and the computed id is passed in.

### Append (`forge ledger append`)
- Inputs: `<epic>` (locates the ledger), `--decision-id` (`^D-\d+$`), `--ticket` (TicketId), `--branch`
  (non-empty). `ts` is generated by the command internally (ISO); there is no required `--ts` flag.
- Calls `appendDecision(file, entry, io)` through an injected `DecisionsLedgerIo` seam.
- Success → print `{ ok: true, ledger: {...} }`, exit 0, byte-deterministic write (existing serialization).
- Failure → print `{ ok: false, code, errors }`, exit 1, **no file write** (every `appendDecision` failure
  path returns before the write).
- Reuses the existing typed codes: `LEDGER_INVALID`, `LEDGER_ENTRY_INVALID`, `LEDGER_SEQUENCE_INVALID`.

## Out of Scope

- Editing `commands/forge-run-ticket.md` — it was just hardened + installed; the prose still passing the
  decision-id flags is harmless (they become cross-checks). A doc cleanup is a separate later follow-up.
- Changing `src/orchestrator/decision-id.ts` (`nextDecisionId` semantics) or
  `src/orchestrator/decisions-ledger.ts` (`appendDecision` / `DecisionsLedgerSchema`) — both are frozen and
  forbidden; this ticket **wires** them, it does not change them. (They are already exported; no edit needed.)
- Any change to `parse-agent pm --expected-decision-id` — it stays the PM echo cross-check
  (`DECISION_ID_MISMATCH`), independent of how the id was assigned.
- Any change to `src/run-report/**` — the run-report takes `decision_id` verbatim from the PM output; no
  schema/assembler/CLI change.
- Any change to `src/orchestrator/packets.ts` — the `PMPacket.inputs.assigned_decision_id` slot already
  exists; the CLI fills it with Core's computed id.

## Acceptance Criteria

1. `forge dispatch pm` reads the decisions ledger from `<epic>/.forge/decisions-ledger.json`.
2. A missing ledger is treated as empty and assigns `D-001`.
3. An existing ledger assigns the next monotonic id using `nextDecisionId`.
4. `--assigned-decision-id` is optional (omitting it still produces a Core-assigned id).
5. `--assigned-decision-id` supplied and equal to Core's computed id → dispatch succeeds.
6. `--assigned-decision-id` supplied and unequal to Core's computed id → `DECISION_ID_PROVENANCE_MISMATCH`,
   and no PM prompt is rendered.
7. `buildPmDispatch` remains pure and performs no file IO.
8. `parse-agent pm --expected-decision-id` is unchanged and still catches a PM echo mismatch
   (`DECISION_ID_MISMATCH`).
9. `forge ledger append <epic> --decision-id <id> --ticket <ticket> --branch <branch>` exists and is routed.
10. `forge ledger append` generates the `ts` internally (no required `--ts`).
11. `forge ledger append` calls `appendDecision`.
12. Append against an absent ledger with `D-001` succeeds as the first entry.
13. A duplicate / lower / gap id append fails via the existing C4 behavior (`LEDGER_SEQUENCE_INVALID`).
14. A malformed existing ledger on append fails (`LEDGER_INVALID`).
15. Any failed append performs no file write.
16. No change to the run-report schema, assembler, or CLI.
17. No change to `commands/forge-run-ticket.md`.
18. No change to agents / charters.
19. No change to `nextDecisionId` semantics or `DecisionsLedgerSchema`.
20. `pnpm test` and `pnpm typecheck` pass.
